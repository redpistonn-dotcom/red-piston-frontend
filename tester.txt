Version    : 1.0 (2026-06-12)
App URL    : https://red-piston.vercel.app  (or http://localhost:5173 for local)

Roles      : PLATFORM_ADMIN (admin console), SHOP_OWNER (ERP), CUSTOMER (marketplace)

HOW TO USE THIS DOCUMENT
- Each test case has: ID, Pre-conditions, Steps, Expected Result.
- Mark each case PASS / FAIL / BLOCKED and note actual behaviour on failure.
- NOTE: backend is on Render free tier — after ~15 min idle the FIRST request
  takes 30–60 seconds (cold start). This is expected; the app must NOT log you
  out during it (see AUTH-20). Wait for it, then continue.
- Email checks: emails are sent from "RedPiston <noreply@notifications.gearitup.me>".
  Check spam folders. Phone-only accounts receive no emails (expected).
- Create fresh throwaway accounts for destructive tests (deletion, rejection).

SECTION 1 — CUSTOMER REGISTRATION & LOGIN
================================================================================

AUTH-01  Customer registration with email
  Pre: not logged in. Steps: Landing page -> Get Started / Create Account ->
  Customer -> Email tab -> enter new email + password (8+ chars) + confirm ->
  Create Account. Expected: account created, taken to "Quick setup" profile
  step (name + I-am-a + optional vehicle), then lands in marketplace.

AUTH-02  Customer registration password rules
  Steps: try password shorter than 8 chars; try mismatched confirm password.
  Expected: inline error for each; no account created.

AUTH-03  Customer registration with existing email
  Steps: register again with an email already used. Expected: error message
  "account already exists... sign in instead" — suggests resume via sign-in.

AUTH-04  Customer profile (name) resume after abandoning setup
  Steps: register a customer, close the tab at the "Quick setup" (name) step
  WITHOUT entering name. Re-open app and Sign In with same email/password.
  Expected: taken straight back to the name step with a green "Welcome back"
  banner — NOT into the app with a blank name.

AUTH-05  Customer registration via phone OTP
  Pre: valid Indian mobile. Steps: Create Account -> Customer -> Phone OTP ->
  enter number -> receive SMS OTP -> enter 6 digits. Expected: account created,
  profile step shown. (OTP comes via Firebase; if SMS does not arrive mark BLOCKED.)

AUTH-06  Customer registration via Google
  Steps: Create Account -> Customer -> Continue with Google -> pick account.
  Expected: account created with Google name/photo prefilled.

AUTH-07  Sign-in with non-existent email
  Steps: Sign In -> Email tab -> enter an email never registered.
  Expected: explicit message "No account found with this email. Please create
  an account first." — NOT a generic invalid-credentials error.

AUTH-08  Sign-in with wrong password (existing account)
  Steps: sign in with valid email + wrong password. Expected: "Invalid email or
  password. N attempt(s) remaining." After 5 failures account locks 30 minutes.

AUTH-09  Sign-in via Google in SIGN-IN mode with no account
  Steps: Sign In (not register) -> Continue with Google with a Google account
  never used here. Expected: "No account found... please create one" — a new
  account must NOT be silently created.

AUTH-10  Forgot password
  Steps: Sign In -> Email tab -> Forgot password -> enter registered email.
  Expected: success message; reset email arrives; link opens reset page; new
  password works; old password stops working; "password changed" email arrives.

AUTH-11  Logout
  Steps: from any logged-in role, log out. Expected: returned to landing/login;
  pressing Back does not re-enter authenticated pages with data.

SECTION 14 — SECURITY / NEGATIVE TESTS
================================================================================

SEC-01  Customer cannot open ERP routes (/dashboard, /inventory, /billing...)
        by typing the URL. Expected: redirected away.
SEC-02  Shop owner cannot open /admin. Expected: blocked.
SEC-03  Pending/rejected shop owner cannot reach the ERP even with an old tab
        open (refresh forces them out).
SEC-04  Invoice/bill/PDF endpoints of ANOTHER shop are not accessible (change
        ids in URLs / API). Expected: 404/forbidden, no data leak.
SEC-05  Marketplace order of another customer/shop not viewable by id
        guessing. Expected: 404.
SEC-06  Negative/zero values rejected: negative payment amount on an invoice,
        zero qty line items, negative discount (treated as 0), adjustment
        below zero stock.
SEC-07  Pagination abuse: API list endpoints with limit=-50 or offset=-100
        do not return reversed/unbounded data (clamped server-side).
SEC-08  No customer phone numbers visible in shop order LISTS; no captain/
        passenger contact data anywhere it shouldn't be.
SEC-09  Session tokens: as_refresh_token in localStorage only; no passwords
        or card data stored client-side (inspect localStorage).
SEC-10  Rate limiting: 6+ rapid OTP requests for the same number get blocked
        with a clear message.
